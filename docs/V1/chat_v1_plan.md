# Chat 模块 V1 方案

## 1. 文档目标

本文件用于明确 `chat` 模块的 V1 开发范围、产品规则、权限边界、前后端协作方式与后续扩展预留。

目标不是完整复刻 QQ，而是在现有 `bbot` + `bbot_server` 体系内，先落地一个可持续迭代的站内即时通讯模块。

## 2. V1 定位

V1 定位为站内 Web IM，核心能力如下：

1. 单聊
2. 群聊
3. 好友关系
4. 文本消息实时收发
5. 历史消息查询
6. 会话列表
7. 未读数
8. WebSocket 即时通知
9. 群创建、加人、退群、基础管理
10. 管理员按权限查看聊天记录
11. 基础搜索

V1 只做文本消息，不做文件消息的发送能力。

## 3. V1 功能范围

### 3.1 必做功能

1. 用户可主动发起单聊
2. 用户可创建群聊
3. 用户可邀请成员加入群聊
4. 用户可退出群聊
5. 会话支持发送纯文本消息
6. 会话支持分页拉取历史消息
7. 会话列表显示最后一条消息、最后活跃时间、未读数
8. 支持好友关系建立、好友列表展示
9. 支持从群成员中发起私聊
10. 支持 WebSocket 即时推送新消息、会话变化、未读数变化
11. 支持用户“删除会话”但不真正删除关系和消息
12. 支持按权限查看聊天记录
13. 支持用户个性化聊天设置落库
14. 支持基础搜索

### 3.2 V1 明确不做

1. 图片消息
2. 文件消息发送
3. 音视频
4. 消息撤回
5. 消息转发
7. 已读回执到单条消息级别
8. 表情包体系
9. 高级禁言策略
10. 群人数限制的实际控制逻辑

## 4. 关键产品规则

### 4.1 删除会话

删除会话不是物理删除。

规则如下：

1. 用户删除会话后，仅将该会话从自己的会话列表中移除
2. 会话消息记录仍然保留
3. 单聊对象仍保留在好友列表中
4. 群聊仍保留在群组列表中
5. 用户后续在好友列表或群组列表中再次打开该会话时，会话重新回到会话列表

因此，V1 需要区分：

1. 会话真实存在
2. 用户是否在自己的会话列表中展示该会话

### 4.2 陌生人私聊规则

允许用户在同一个群里，对不是好友的群成员发起私聊。

但需要限制消息类型：

1. 若双方不是好友，则仅允许发送文本消息
2. 若未来支持图片、文件等消息类型，则发送时需要先校验好友关系
3. 非好友尝试发送非文本消息时，前后端都要阻止
4. 阻止时提示：请先添加对方为好友

V1 虽然只做文本消息，但这一规则必须在模型和服务层预留，以便后续扩展文件/图片消息时不返工。

### 4.3 群创建规则

1. 允许任意用户创建群聊
2. 群聊人数限制字段需要预留，但 V1 不做人数上限控制逻辑
3. 群创建者默认为群主

### 4.4 未读数规则

V1 采用会话级未读数，不做单条消息级已读回执。

规则如下：

1. 每个用户在每个会话下维护自己的未读计数
2. 打开会话并完成消息同步后，当前会话未读数清零
3. 会话列表展示未读数
4. 后续如需扩展已读回执，再单独增加消息阅读状态设计

### 4.5 好友申请审批规则

好友关系必须经过审批后才能正式建立。

规则如下：

1. 用户发起好友申请后，对方需要明确同意，双方才成为好友
2. 若对方拒绝，则本次申请结束，不建立好友关系
3. 若双方已经是好友，不能重复申请
4. 若双方存在待处理的相反方向好友申请，则直接自动通过，双方立即成为好友
5. 自动通过时，需要将两条申请记录同步更新为已通过或以等价状态收口，避免保留悬空待处理记录

### 4.6 群入群审批规则

群聊需要有独立配置，用于控制是否开启入群审批。

规则如下：

1. 每个群聊有自己的配置项
2. 配置项中包含“加入群聊是否需要审批”开关
3. 当关闭审批时，符合邀请逻辑的用户可直接加入
4. 当开启审批时，群友邀请好友进群后，需要群主或群管理员审批
5. 只有审批通过后，目标用户才正式成为群成员
6. 被拒绝后，不加入群聊
7. 群主默认有审批权限，群管理员也有审批权限，普通成员没有审批权限

### 4.7 超级管理员群聊行为规则

一般情况下，超级管理员在聊天系统中的普通会话行为，与普通用户一致。

规则如下：

1. 超级管理员可以像普通成员一样被踢出群聊
2. 超级管理员可以像普通成员一样被禁言
3. 超级管理员不会因为后台身份而天然豁免群规则

### 4.8 超级管理员隐身巡检规则

需要为超级管理员增加一个“隐身巡检会话”的能力，该能力通过超级管理员设置项控制。

规则如下：

1. 该能力仅超级管理员可见、可配置
2. 开启后，超级管理员的会话列表中直接出现系统内所有会话
3. 超级管理员可以进入任意会话查看消息记录
4. 超级管理员进入这类会话时，不作为真实成员加入群聊
5. 群聊成员人数不会因此变化
6. 超级管理员不能在隐身巡检状态下发送消息
7. 其他用户不会感知超级管理员进入了该会话
8. 该能力本质上是“只读旁观”，不是隐式加群

实现上建议与普通成员关系分离，不要把该状态直接建模成真实 `ChatConversationMember`。

### 4.9 基础搜索规则

V1 需要提供基础搜索能力。

规则如下：

1. 用户输入关键字后，搜索结果以分层下拉形式展示
2. 搜索范围至少包括会话名、用户名、聊天消息记录
3. 搜索结果需要分组展示，例如“会话”“用户”“消息”
4. 点击会话结果时，直接跳转到对应会话
5. 点击用户结果时，跳转到对应用户单聊或其资料入口
6. 点击消息结果时，跳转到对应会话，并尽量定位到该消息所在位置

V1 的搜索以可用为主，不要求做复杂全文检索方案，但数据结构和接口应支持后续增强。

## 5. 好友关系设计

V1 增加好友体系，但保持简化。

建议包含以下能力：

1. 发送好友申请
2. 接受好友申请
3. 拒绝好友申请
4. 删除好友
5. 好友列表展示
6. 从好友列表打开单聊
7. 双向待处理申请自动互相通过

V1 不做：

1. 好友分组
2. 好友备注历史
3. 黑名单
4. 特别关心

补充规则：

1. 好友申请是显式审批制，不是单向提交即生效
2. 若检测到双方互相申请，系统自动通过，无需再次人工审批

## 6. 消息类型设计

V1 只开放 `text`。

但消息模型建议从一开始就预留类型字段，便于后续扩展。

建议消息类型枚举预留：

1. `text`
2. `system`
3. `image`
4. `file`

当前 V1 仅允许：

1. `text`
2. `system`

## 7. 文件消息扩展预留

V1 不做文件消息，但未来文件消息需要复用现有上传系统。

预留规则如下：

1. 每个用户在自己的专属目录下新增一个不可删除目录，用于存放聊天产生的文件
2. 该目录由系统自动创建
3. 普通用户不可删除、不可重命名、不可移动该目录
4. 文件消息未来优先复用现有上传、分片、合并和存储逻辑

建议未来目录命名为：

1. `chat_files`

该目录设计只做预留，V1 不实际接入聊天发送流程。

## 8. 权限体系设计

### 8.1 普通权限

建议新增一组 `chat.*` 权限，例如：

1. `chat.view_conversation`
2. `chat.send_message`
3. `chat.create_group`
4. `chat.manage_group`
5. `chat.add_friend`
6. `chat.review_all_messages`

### 8.2 全量聊天记录查看权限

需要新增一个“查看所有聊天记录”的权限。

规则如下：

1. 权限编码建议为 `chat.review_all_messages`
2. 只有超级管理员可以创建、分配、回收带有该权限的角色
3. 拥有该权限的用户可以查看所有聊天记录
4. 不带该权限的用户只能查看自己参与的会话和消息
5. 带有该权限的角色，对非超级管理员不可见、不可操作
6. 例如系统管理员角色，即使具备较高后台权限，也不能看到或编辑该权限，除非其本身是超级管理员

这意味着除了普通 RBAC 规则，还需要在角色管理和权限管理层增加“超级管理员专属可见性”约束。

## 9. 用户个性化设置

当前前端设置是每个用户独立的，V1 需要把设置落库。

建议新增用户配置表，例如：

1. `UserPreference`

建议包含但不限于以下内容：

1. 主题模式
2. 布局偏好
3. 聊天页配置
4. 会话排序偏好
5. 消息通知开关
6. 是否免打扰
7. 其他个性化设置 JSON 扩展字段

建议：

1. 与用户一对一关联
2. 提供默认值
3. 使用结构化字段 + JSON 扩展字段组合

后续前端设置页的数据来源从本地持久化逐步迁移到数据库。

## 10. 数据模型草案

以下为建议模型，名称可按项目风格调整。

### 10.1 好友关系

#### `ChatFriendRequest`

用于好友申请。

建议字段：

1. `from_user`
2. `to_user`
3. `status`，如 `pending / accepted / rejected / canceled`
4. `message`
5. `handled_at`
6. `created_at`
7. `updated_at`

#### `ChatFriendship`

用于表示好友关系。

建议字段：

1. `user`
2. `friend`
3. `source`
4. `created_at`

建议使用双向两条记录，简化查询。

### 10.2 会话

#### `ChatConversation`

建议字段：

1. `type`，如 `direct / group`
2. `name`
3. `avatar`
4. `owner`
5. `max_members`，预留字段
6. `is_active`
7. `last_message`
8. `last_message_at`
9. `created_at`
10. `updated_at`

单聊会话建议全局唯一，避免同一对用户出现多个 direct 会话。

### 10.2.1 群配置

#### `ChatGroupConfig`

建议为群聊增加独立配置表，避免将群规则字段过度堆叠在会话主表。

建议字段：

1. `conversation`
2. `join_approval_required`
3. `max_members`，预留字段
4. `allow_member_invite`
5. `created_at`
6. `updated_at`

其中：

1. `join_approval_required` 用于控制拉人入群是否需要审批
2. `max_members` 先做预留，V1 不做实际人数限制逻辑

### 10.3 会话成员

#### `ChatConversationMember`

建议字段：

1. `conversation`
2. `user`
3. `role`，如 `owner / admin / member`
4. `joined_at`
5. `is_muted`
6. `is_pinned`
7. `show_in_list`，控制会话是否显示在当前用户会话列表中
8. `unread_count`
9. `last_read_message_id`
10. `last_read_at`
11. `last_delivered_message_id`
12. `extra_settings`

V1 至少需要 `show_in_list` 和 `unread_count`。

对于超级管理员隐身巡检能力，不建议通过新增普通成员记录来表达，避免污染真实群成员关系、未读数和人数统计。

### 10.4 消息

#### `ChatMessage`

建议字段：

1. `conversation`
2. `sender`
3. `message_type`
4. `content`
5. `payload`
6. `is_system`
7. `created_at`
8. `updated_at`
9. `revoked_at`

V1 中：

1. `message_type` 只允许 `text` 和 `system`
2. `content` 为主要展示文本
3. `payload` 预留未来扩展

### 10.5 用户配置

#### `UserPreference`

建议字段：

1. `user`
2. `theme_mode`
3. `settings_json`
4. `created_at`
5. `updated_at`

建议在 `settings_json` 中预留聊天相关配置，例如：

1. 超级管理员隐身巡检开关
2. 会话列表排序偏好
3. 搜索历史

## 11. 接口分层建议

建议遵循：

1. HTTP 负责数据查询、初始化、管理操作
2. WebSocket 负责实时消息与即时通知

不要把会话创建、好友申请、群成员管理等重业务逻辑全部压进 WebSocket。

### 11.1 HTTP 接口建议

建议至少包含以下资源：

1. 好友申请
2. 好友列表
3. 会话列表
4. 会话详情
5. 历史消息
6. 创建单聊
7. 创建群聊
8. 邀请成员
9. 退出群聊
10. 删除会话
11. 群审批处理
12. 基础搜索
13. 用户设置
14. 管理员聊天记录检索

### 11.2 WebSocket 事件建议

客户端发送：

1. `send_message`
2. `mark_read`
3. `typing`
4. `ping`

服务端推送：

1. `new_message`
2. `message_ack`
3. `conversation_updated`
4. `unread_updated`
5. `friend_request_updated`
6. `friendship_updated`
7. `system_notice`
8. `typing`
9. `group_join_request_updated`

## 12. WebSocket 即时消息通知工具

前后端需要配合做一个统一工具，专门处理 WebSocket 的即时消息通知。

该工具建议承担以下职责：

1. 建立和维护连接
2. 心跳保活
3. 自动重连
4. 统一事件分发
5. 按会话订阅和派发消息
6. 未读数同步
7. 好友申请和系统通知分发
8. 统一错误处理
9. 群审批状态分发

### 12.1 前端职责

建议在前端形成独立聊天实时层，例如：

1. 连接管理器
2. 事件总线或统一分发器
3. 消息缓存协调
4. 当前会话阅读状态同步

### 12.2 后端职责

建议在后端形成统一消息通知入口，例如：

1. 推送新消息到目标用户组
2. 推送会话列表刷新事件
3. 推送未读数更新
4. 推送好友关系变更
5. 推送系统通知

建议尽量避免业务代码直接散落调用 channel layer，统一走事件封装层。

## 13. 前端页面建议

V1 页面建议包含：

1. 会话列表页
2. 好友列表页
3. 群组列表页
4. 聊天主窗口
5. 群信息侧栏
6. 新建群聊弹窗
7. 好友申请处理弹窗或抽屉
8. 搜索下拉面板
9. 群入群审批处理面板

聊天主界面建议结构：

1. 左栏：好友、群组、会话切换区
2. 中栏：消息区与输入框
3. 右栏：会话详情或群成员信息

## 15. 当前已确认决策汇总

1. chat 模块按 V1 范围推进
2. V1 包含好友关系
3. V1 只做文本消息
4. 未读数按会话级处理
5. 文件消息未来复用现有上传系统
6. 用户删除会话时仅从自己的会话列表移除
7. 同群非好友之间允许私聊，但未来非好友仅允许文本消息
8. 任意用户可创建群聊
9. 群人数限制先预留字段，后续再做逻辑
10. 需要单独的聊天记录全局查看权限，并且该权限及对应角色仅超级管理员可见可控
11. 用户个性化设置改为数据库持久化
12. 好友申请需要审批，双方互相申请时自动通过
13. 群聊有独立配置，开启审批后拉人入群需群主或群管理员审批
14. 一般情况下超级管理员在群里按普通用户处理，可被踢出、禁言
15. 超级管理员可通过专属设置开启隐身巡检，查看系统所有会话但不能发言且不影响群成员状态
16. V1 增加基础搜索，搜索会话名、用户名和消息内容，并支持跳转定位
