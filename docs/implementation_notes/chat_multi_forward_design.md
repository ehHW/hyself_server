# Chat 多选合并转发技术实现思路

## 1. 文档目标

本文档说明当前项目中“多选消息后进行合并转发”的完整制作思路，重点覆盖以下内容：

1. 为什么要把它设计成正式消息类型，而不是前端临时拼装卡片
2. 前端如何组织多选、目标会话选择、转发方式选择与聊天记录查看
3. 后端如何校验、组装、落库、广播并复用现有消息链路
4. 附件、嵌套聊天记录、WebSocket 实时更新如何协同工作
5. 当前实现的边界、测试策略和后续扩展方向

本文是“技术实现思路文档”，不是用户使用说明，因此会直接对应当前代码实现。

## 2. 功能目标与产品约束

### 2.1 功能目标

多选转发的目标不是简单批量复制，而是支持两种语义完全不同的转发模式：

1. 逐条转发：把选中的每条消息，按当前发送者身份重新发送到目标会话
2. 合并转发：把选中的多条消息封装成一条新的聊天记录消息，由接收方点击后再展开查看

### 2.2 关键产品约束

当前实现遵守以下规则：

1. 单条转发默认走逐条转发，不显示“转发自 xx 用户”
2. 合并转发至少需要 2 条消息
3. 合并转发产物必须是正式消息，能像普通消息一样进入消息列表、排序、同步和广播
4. 聊天记录内部允许再次包含聊天记录，形成递归嵌套
5. 附件消息不能只复制文件 URL，必须复用资产引用链路，保证权限和来源可追踪
6. 被转发消息必须来自当前用户可访问的会话，目标会话也必须允许当前用户发送

这几个约束直接决定了本功能必须以“消息领域能力”落地，而不是只做前端界面效果。

## 3. 为什么采用 chat_record 正式消息类型

### 3.1 不采用前端临时卡片的原因

如果只让前端在多选后拼一个卡片文本发送，会有几个明显问题：

1. 后端无法理解该消息的业务含义，搜索、序列化、预览都只能当普通文本处理
2. WebSocket 广播只会传播文本，其他端无法稳定还原卡片结构
3. 聊天记录里的附件、嵌套记录等结构会丢失
4. 后续做导出、检索、审核时没有稳定数据结构可依赖

### 3.2 当前方案

当前项目通过为 `ChatMessage` 增加 `chat_record` 消息类型，把合并转发产物纳入统一消息系统。

对应位置：

1. 模型定义在 [bbot_server/chat/models.py](bbot_server/chat/models.py)
2. 合并转发编排在 [bbot_server/chat/application/commands/forwarding.py](bbot_server/chat/application/commands/forwarding.py)
3. 前端类型声明在 [bbot/src/types/chat.ts](bbot/src/types/chat.ts)

这样做的好处是：

1. 消息列表、最近消息预览、已读未读、会话排序都能天然复用已有基础设施
2. 新消息广播沿用 `chat.message.created`
3. 前端只要认识 `chat_record` 类型，就能稳定渲染和递归展开
4. 后续如果要支持导出、审核或全文索引，可以直接基于结构化 payload 扩展

## 4. 整体架构分层

## 4.1 前端职责

前端只负责交互组织和结构化展示，不负责决定最终业务真实性。

当前核心前端职责包括：

1. 多选状态管理
2. 选择目标会话或好友
3. 在多条消息时弹出“逐条转发 / 合并转发”选择
4. 调用转发 API 并传递 `forward_mode`
5. 在消息列表中识别 `chat_record` 并渲染卡片
6. 打开聊天记录查看器，并支持递归查看嵌套聊天记录

## 4.2 后端职责

当前核心后端职责包括：

1. 校验消息 ID 列表与目标会话是否合法
2. 校验用户是否有权访问来源会话及目标会话
3. 根据 `forward_mode` 执行逐条转发或合并转发
4. 为合并转发组装 `chat_record` 结构化 payload
5. 创建正式消息记录并广播给当前会话成员
6. 维护会话预览、未读计数和会话列表展示状态

## 5. 交互流程设计

### 5.1 用户侧操作路径

多选合并转发在交互上分成四步：

1. 用户在消息列表中进入多选模式并勾选多条消息
2. 点击转发，前端先打开“选择聊天”弹窗，确定目标会话
3. 如果只选中 1 条消息，则直接按逐条转发发送
4. 如果选中多条消息，则再弹出“逐条转发 / 合并转发”二次选择

这个拆分的原因是：

1. 用户必须先明确“发给谁”
2. 只有多条消息时，才存在逐条和合并的分岔
3. 把目标选择和转发模式选择拆开，能减少一个弹窗里承载的决策密度

### 5.2 前端状态组织

`MessageWorkspace.vue` 里与转发相关的状态大致分为：

1. 当前是否处于多选模式
2. 已选消息 ID 列表
3. 转发目标选择弹窗状态
4. 转发模式选择弹窗状态
5. 当前待确认的目标会话
6. 最近转发目标缓存
7. 聊天记录查看器弹窗栈

这样设计的目的不是把所有逻辑都堆在一个对象里，而是围绕用户动作顺序组织状态机：

1. 选消息
2. 选会话
3. 选模式
4. 发送结果
5. 查看聊天记录

## 6. 前端实现拆解

### 6.1 多选入口与消息选择

多选入口挂在消息右键菜单中，进入后消息行会出现选择控件。当前做法是：

1. 右键某条消息，选择“多选”
2. 前端切换 `messageSelectionMode`
3. 之后点击消息行即可切换选中状态
4. 顶部出现多选操作栏，显示已选数量与“转发 / 取消多选”动作

这里的设计重点不是“复选框样式”，而是让多选状态仍然停留在消息工作区内部，避免切换页面级状态。

### 6.2 目标会话选择弹窗

目标会话弹窗提供两类选择源：

1. 最近转发目标
2. 最近聊天和好友

这样设计是为了同时覆盖两类使用场景：

1. 高频转发给固定对象
2. 临时从最近聊天里找目标

如果目标是好友但尚未存在单聊，会先尝试创建或打开单聊，再继续完成后续转发流程。

### 6.3 转发模式弹窗

当前只在选中多条消息时展示二次弹窗，原因如下：

1. 单条消息没有“合并”的必要
2. 如果每次都让用户选模式，会明显增加操作负担
3. 单条转发当前产品语义就是“以当前用户身份正常发送”

模式文案对应如下：

1. 逐条转发：每条消息重新发送，不保留“转发自”标识
2. 合并转发：把多条消息打包为一条聊天记录消息

### 6.4 聊天记录卡片与查看器

当前前端把 `chat_record` 拆成两个展示层：

1. `ChatRecordCard.vue`：用于消息列表中的摘要卡片
2. `ChatRecordViewer.vue`：用于弹窗中查看内部明细

这样拆分的原因是：

1. 消息列表需要轻量摘要，不适合直接展开全部内容
2. 查看器需要支持滚动、递归打开、右键菜单、附件点击等交互
3. 递归嵌套时，卡片和查看器天然可以互相复用

### 6.5 查看器内的递归能力

聊天记录内部的某一项如果本身又是 `chat_record`，当前不会强行拍平成普通文本，而是保留原始结构：

1. 后端把嵌套记录序列化到 `item.chat_record`
2. 前端查看器识别后继续渲染 `ChatRecordCard`
3. 用户点击嵌套卡片，会再打开下一层记录查看器

这保证了“聊天记录转发聊天记录”的信息不会丢失。

### 6.6 附件在聊天记录中的展示

聊天记录里如果包含附件消息，查看器会根据资源类型分别处理：

1. 图片直接预览缩略图
2. 音视频允许新窗口打开预览
3. 非预览型文件直接触发下载

这样做的原因是：

1. 用户在聊天记录里看到的应该仍然是“消息内容”，而不是一串 URL
2. 非预览型文件没必要先打开浏览器页面再下载
3. 处理策略与主聊天窗口保持一致，减少认知落差

## 7. 后端实现拆解

### 7.1 API 入参设计

转发接口通过 `ForwardMessagesSerializer` 接收三个关键字段：

1. `target_conversation_id`
2. `message_ids`
3. `forward_mode`

其中 `forward_mode` 当前只允许两种值：

1. `separate`
2. `merged`

这个设计的优点是：

1. 接口保持单一入口，前端不需要分别调两个地址
2. 逐条和合并共享同一套来源校验、目标校验与权限校验
3. 后续如果要扩展“带批注转发”“引用式转发”，可以继续用模式字段扩展

### 7.2 命令入口

实际业务编排从 `execute_forward_messages_command` 开始。

该命令先做统一前置校验：

1. `message_ids` 不能为空
2. `forward_mode` 必须合法
3. 目标会话必须存在且状态有效
4. 当前用户对目标会话必须具有发送权限
5. 所有来源消息必须存在、不是系统消息、且用户有访问权限

这一层的目标是：无论最终走逐条还是合并路径，都先把非法输入挡在外面。

### 7.3 逐条转发路径

逐条转发的核心思路是“把原消息转换成新的发送动作”，而不是复制数据库记录。

具体分支如下：

1. 文本消息：复用 `execute_send_text_message_command`
2. 附件消息：复用 `execute_send_asset_message_command`
3. 聊天记录消息：直接创建新的 `chat_record` 消息，复用已有 payload

这样做有两个核心价值：

1. 新消息会重新走当前发送链路，保证预览、未读数、广播逻辑一致
2. 附件消息通过 `source_asset_reference_id` 复用资产链路，避免脱离资源域

### 7.4 合并转发路径

合并转发的关键步骤是：

1. 把选中的消息按创建时间、序号和 ID 排序
2. 为每条消息生成一个可嵌套的聊天记录 item
3. 根据来源会话和发送者组合生成记录标题
4. 组装完整 `chat_record` payload
5. 创建一条新的 `ChatMessage`，其 `message_type = chat_record`
6. 广播新消息事件

这里最关键的不是“生成卡片”，而是“构造一个稳定的中间数据结构”。

## 8. chat_record 数据结构设计

当前聊天记录 payload 结构如下：

```json
{
  "version": 1,
  "title": "群聊的聊天记录",
  "footer_label": "聊天记录",
  "items": [
    {
      "source_message_id": 101,
      "sequence": 9001,
      "conversation_id": 12,
      "message_type": "text",
      "sender_name": "Alice",
      "sender_avatar": "/media/avatars/a.png",
      "content": "你好"
    }
  ]
}
```

### 8.1 顶层字段说明

1. `version`
   用于未来结构演进时做兼容
2. `title`
   用于消息列表里的卡片标题和查看器标题
3. `footer_label`
   当前固定为“聊天记录”，用于卡片底部标识
4. `items`
   真实消息条目数组

### 8.2 item 字段说明

每个 item 都保留“足够展示，但不过度冗余”的字段：

1. `source_message_id`
   用于定位来源消息
2. `sequence`
   保留原消息序号，便于未来扩展跳转或排序审计
3. `conversation_id`
   保留来源会话信息
4. `message_type`
   决定前端如何渲染
5. `sender_name`、`sender_avatar`
   用于查看器中的消息头展示
6. `content`
   文本内容或默认展示文案
7. `asset`
   附件消息附带的资产信息
8. `chat_record`
   如果当前条目本身就是聊天记录，则递归嵌套

### 8.3 为什么保留递归结构

不把嵌套记录打平的原因是：

1. 打平会丢失原有上下文边界
2. 被再次转发的聊天记录本身就是一个完整对象
3. 递归结构能让前端更自然地实现“层层打开”的查看体验

## 9. 标题与摘要生成策略

### 9.1 合并记录标题

当前标题规则在后端统一生成：

1. 来源是群聊时，标题固定为“群聊的聊天记录”
2. 来源是单聊且只涉及一个发送者时，标题为“某某的聊天记录”
3. 来源是单聊且涉及两个发送者时，标题为“甲和乙的聊天记录”
4. 来源发送者超过两人时，标题为“甲和乙等的聊天记录”

这样做的目的不是绝对精确，而是在不暴露过多复杂元数据的情况下，给接收方一个足够清晰的上下文摘要。

### 9.2 会话预览

为了让会话列表可读，后端还会为 `chat_record` 消息生成预览文案。

当前做法是：

1. 普通文本取消息内容
2. 附件消息取 `display_name`
3. 聊天记录消息取 payload 内的 `title`

因此合并转发发送完成后，会话列表最近消息不会显示成空白或 JSON，而是显示“某某的聊天记录”一类可读摘要。

## 10. 事件广播与同步机制

### 10.1 为什么不单独造一套“转发广播”

当前项目没有为转发再设计新的 WebSocket 事件类型，而是复用普通消息广播链路。

原因很直接：

1. 对接收方来说，转发完成后的结果就是“会话里新增了一条消息”
2. 如果另起一套事件，会让前端消费分支变复杂
3. 统一复用 `chat.message.created`、`chat.conversation.updated`、`chat.unread.updated` 更稳定

### 10.2 广播过程

在后端 `_emit_message_events` 中，发送后会完成这些动作：

1. 序列化新消息
2. 重新序列化当前会话
3. 对发送方推送新消息与会话更新
4. 对其他活跃成员推送新消息、会话更新和未读更新
5. 如果目标会话原本被隐藏，还会自动恢复显示

也就是说，合并转发虽然是新功能，但在“消息落库后的传播”这一段，完全走了现有聊天域的标准基础设施。

## 11. 权限与安全考虑

### 11.1 来源权限校验

每条来源消息都必须属于当前用户可访问的会话。

这样可以避免：

1. 用户构造任意 message_id 窃取他人消息内容
2. 越权把本不应看到的消息转发到自己可见会话

### 11.2 目标权限校验

目标会话必须满足：

1. 会话存在且处于活动状态
2. 当前用户是成员
3. 当前用户当前时刻允许发送消息

这会自动继承现有群禁言、成员状态、隐藏会话恢复等规则。

### 11.3 附件权限与来源追踪

附件消息不直接复制 URL，而是依赖 `source_asset_reference_id` 再走发送附件命令。

这么做的好处是：

1. 附件来源可追踪
2. 资产域和聊天域关系不会断裂
3. 后续做审计、清理或权限策略时，不会因为“裸 URL 转发”而失控

## 12. 与回复、预览、附件体系的兼容

### 12.1 回复预览兼容

因为系统新增了 `chat_record` 类型，所以发送文本或附件时，如果要带回复引用，回复预览逻辑也必须认识这种新类型。

当前实现已经同步更新了回复预览构造逻辑，使其在引用聊天记录时也能生成正确摘要。

### 12.2 消息列表兼容

前端消息列表对消息类型的判断已经扩展为：

1. 文本消息
2. 图片 / 文件消息
3. 群邀请消息
4. 聊天记录消息

因此 `chat_record` 不会破坏原有消息渲染分支，而是作为新的显示分支自然接入。

## 13. 测试策略

当前项目已经为该能力补充了针对性测试，主要覆盖：

1. 文本消息可以转发到目标会话
2. 多条消息合并转发后会创建 `chat_record` 消息
3. 聊天记录消息再次逐条转发时，能够复用已有 `chat_record` payload
4. 相关消息类型与预览逻辑能正常工作

测试代码位于 [bbot_server/chat/tests.py](bbot_server/chat/tests.py)。

测试重点不是单纯断言接口 200，而是验证：

1. 新消息类型是否正确
2. payload 是否包含预期结构
3. 转发结果是否进入目标会话
4. 递归聊天记录是否可继续被转发

## 14. 当前实现的优点

当前方案的核心优点可以概括为四点：

1. 业务真实：合并转发是正式消息，不是前端临时 UI
2. 链路复用：落库、预览、广播、未读都复用现有系统
3. 结构稳定：`chat_record` payload 支持文本、附件和递归嵌套
4. 易于扩展：未来增加导出、审核、搜索时有明确结构基础

## 15. 当前实现的局限

虽然当前方案已经可用，但仍有一些可以继续增强的点：

1. `chat_record` 目前是 JSON 结构，没有单独拆成数据库子表
2. 内部 item 暂未记录更细粒度的来源上下文，例如来源会话名称
3. 合并转发标题规则偏摘要化，不是完整语义描述
4. 当前主要依赖消息内嵌 payload，后续若要做复杂检索可能需要额外索引层
5. 文档层面还没有配套时序图和异常流矩阵

这些局限不影响当前可用性，但会影响后续深度治理能力。

## 16. 后续扩展建议

建议后续按以下方向演进：

1. 为 `chat_record` 增加显式 schema 版本升级策略
2. 给聊天记录 item 增加 `conversation_name` 等展示字段，减少前端兜底逻辑
3. 增加“合并转发附言”能力，让用户在聊天记录卡片前附加说明
4. 为聊天记录导出、复制摘要、内容搜索建立专门应用服务
5. 在文档体系中补充配套时序图、错误码和事件流说明

## 17. 结论

多选合并转发的核心设计思想，不是“把多条消息塞进一个弹窗”，而是：

1. 把批量消息抽象为一种正式的可传输消息对象
2. 用结构化 payload 表达内部内容
3. 让前端负责交互和递归展示
4. 让后端负责真实性、权限、持久化和广播

因此，当前实现既满足产品体验，也保持了聊天系统架构的一致性。这也是本功能最重要的技术价值。
