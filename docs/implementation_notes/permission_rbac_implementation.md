# 权限与角色实现说明

本文档说明当前项目权限系统的落地方式、角色基线、运行时同步逻辑，以及本地重置后的初始化结果。

## 1. 设计目标

当前权限实现的目标有四个：

1. 让菜单可见性、接口访问控制、用户角色管理使用同一套权限编码。
2. 正式运行角色保持收敛，只保留三个长期角色：超级管理员、系统管理员、普通用户。
3. 浏览器烟雾测试账号与角色不常驻，改为按需创建、测试后删除。
4. 本地开发环境支持一键重置，避免残留历史账号、上传文件和错误权限状态。

## 2. 核心模型

权限相关核心模型位于 user 应用：

1. Permission：权限编码、名称、描述。
2. Role：角色名、描述，以及角色与权限的多对多关系。
3. User：用户与角色的多对多关系，超级管理员额外通过 is_superuser 标识获得全量能力。

其中 User 同时继承 Django 的 AbstractUser，因此认证、密码哈希和登录态仍复用 Django 原生体系。

## 3. 权限编码来源

默认权限定义集中在 user/signals.py 的 DEFAULT_PERMISSIONS。

这些编码既服务后端接口校验，也服务前端菜单与页面显隐。

当前主要按业务域分组：

1. user.\*：用户、角色、权限管理。
2. file.\*：资源中心查看、上传、回收站、系统资源管理。
3. chat.\*：聊天查看、发消息、发附件、好友、转发、撤回、巡检。
4. game.\*：排行榜与成绩提交。
5. entertainment.\*：音乐、视频入口展示。

## 4. 三个正式角色

当前正式角色只有三个：

1. 超级管理员
2. 系统管理员
3. 普通用户

它们的职责边界如下。

### 4.1 超级管理员

超级管理员是系统内置最高权限角色，同时对应 Django 的 is_superuser。

特点：

1. 拥有全部 Permission。
2. 可操作超级管理员专属能力。
3. 登录后会自动绑定到超级管理员角色。
4. 普通角色分配流程禁止把该角色直接分配给一般用户。

### 4.2 系统管理员

系统管理员是后台主运营角色。

特点：

1. 拥有除超级管理员专属权限外的所有默认权限。
2. 不是 is_superuser。
3. 能处理大部分后台与业务管理动作。
4. 无法越过“超级管理员专属”限制。

当前超级管理员专属权限为：

1. chat.review_all_messages

如果未来增加更多超管专属编码，应同时加入 user/signals.py 中的 SUPER_ADMIN_ONLY_PERMISSION_CODES，并同步前端与接口限制。

### 4.3 普通用户

普通用户是默认基础角色。

特点：

1. 新建普通用户时，若未显式指定角色，会自动分配普通用户角色。
2. 普通用户覆盖系统的基础使用能力，例如资源中心、聊天、娱乐中心的常规访问。
3. 用于保证“每个非超管用户至少有一个角色归属”。

## 5. 启动与同步机制

正式角色和默认权限通过 user/signals.py 中的 ensure_default_permissions_synced 统一维护。

该同步逻辑在以下场景会被调用：

1. post_migrate：迁移完成后自动补齐默认权限与正式角色。
2. 登录、资料读取、用户列表等入口：运行时再次兜底，避免本地开发时角色缺失。
3. 新权限创建后：自动把新增权限绑定给超级管理员角色。

同步原则是：

1. 默认权限缺失则补齐。
2. 正式角色缺失则创建。
3. 已被软删除的正式角色会自动恢复。
4. 超级管理员角色始终绑定全部权限。
5. 系统管理员与普通用户按既定权限集重新对齐。

## 6. 运行时访问控制

后端访问控制有两层。

### 6.1 接口层

接口层主要使用两种方式：

1. ViewSet 的 ActionPermission：把 action 映射到权限编码。
2. ensure_request_permission：在普通 APIView 中直接校验指定权限。

这保证了接口是否可访问由权限编码直接驱动，而不是散落在每个视图中的条件分支。

### 6.2 业务层

业务层仍保留更细的保护规则，例如：

1. 超级管理员用户不能被普通流程修改或删除。
2. 当前登录用户不能停用自己或删除自己。
3. 删除角色时不能导致用户失去全部角色。
4. 某些聊天群内管理能力除了权限编码，还要求当前用户在该群里的成员角色满足条件。

因此，权限编码负责“是否具备入口资格”，业务规则负责“是否满足具体对象级约束”。

## 7. 前端权限上下文

前端不直接自己推导权限集合，而是通过 auth/permission-context 接口获取权限上下文。

当前接口返回两类数据：

1. permission_codes：当前用户持有的权限编码列表。
2. visible_menu_keys：后端按规则计算后的可见菜单键集合。

这样做的好处是：

1. 菜单显隐和后端权限语义保持一致。
2. 前端不需要复制一份复杂 RBAC 推导逻辑。
3. 角色或权限变化后，可以通过 WebSocket 事件触发前端刷新上下文。

## 8. 烟雾测试账号策略

浏览器烟雾测试仍然使用专门账号，但这些账号不再视为正式角色体系的一部分。

当前策略是：

1. seed_smoke_data 按需创建测试账号与临时角色。
2. 临时角色只用于验证权限矩阵，不参与正式角色基线。
3. cleanup_smoke_data 在测试完成后删除这些账号、临时角色和共享群聊。
4. reset_local_data 也会一并清除所有这类测试数据。

这样可以避免固定密码的测试超管长期残留在本地数据库中。

## 9. 本地重置命令

为了让开发环境回到确定状态，新增了 reset_local_data 管理命令。

该命令会执行以下步骤：

1. flush 数据库，清空现有业务数据。
2. 清空 uploads 目录下的全部文件与子目录。
3. 重新同步默认权限与三个正式角色。
4. 创建 admin 与 5 个普通用户。

默认创建结果：

1. admin：超级管理员账号。
2. user01 到 user05：普通用户账号。
3. 超级管理员、系统管理员、普通用户：三个正式角色。

命令会在终端输出本次生成的密码，避免项目里长期写死默认口令。

## 10. 长度限制调整

本轮还同步调整了账号字段的长度策略：

1. 登录序列化层不再额外限制用户名和原始密码长度。
2. 昵称改为 TextField，不再设置应用层最大长度。
3. 用户名列扩展到 255，以兼顾数据库唯一索引约束与更宽松的输入范围。
4. 密码最小长度校验已从 Django 默认校验器中移除。

说明：

1. 原始密码输入在业务层不再设置固定长度上限。
2. 用户名由于数据库唯一索引约束，仍然需要有限字段宽度；当前实现已放宽到 255，而不是无限文本列。

## 11. 后续建议

如果后续权限系统继续扩展，建议保持以下约束：

1. 新权限必须先进入 DEFAULT_PERMISSIONS，再接入接口与前端。
2. 正式长期角色数量保持收敛，避免重新膨胀成大量常驻角色。
3. 测试账号与正式账号严格隔离，所有测试数据都必须可清理。
4. 每次新增角色边界时，同时更新本目录文档与回归测试。
